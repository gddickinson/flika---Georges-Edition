# AI Tools

Flika integrates several AI/ML-powered tools for image analysis and automation,
accessible from the **AI** menu and the **Simulation** menu. Some tools require
additional dependencies (PyTorch, specific model packages) that are installed on
demand. The Claude-powered features require an Anthropic API key.

---

## AI Menu Overview

| Menu Item | Description |
|---|---|
| **Claude > Live Session** | Agentic mode -- Claude directly executes actions in flika |
| **Claude > Generate Script** | Multi-turn AI script generation from natural language |
| **Claude > Generate Plugin** | AI plugin scaffolding with preview, validation, and auto-load |
| **Claude > Safety Settings** | Configure permissions for AI-generated code execution |
| **AI Denoiser** | Deep learning image denoising |
| **Pixel Classifier** | Train and apply pixel-level classifiers |
| **Particle Localizer** | AI-based particle detection |
| **Segmentation > Cellpose** | Cell segmentation with Cellpose |
| **Segmentation > StarDist** | Star-convex polygon cell detection |
| **Segmentation > SAM Interactive** | Segment Anything Model interactive segmentation |
| **BioImage.IO Model Zoo** | Browse and load community models |

## Claude Live Session

**AI > Claude > Live Session**

The most powerful AI feature in flika. Live Session is an **agentic mode** where
Claude can directly inspect your data and execute image processing operations
without you writing any code. You describe what you want in natural language, and
Claude decides which tools to call, executes code, checks the results, and
continues until the task is complete.

### How It Works

Live Session uses the Anthropic **tool-use API**. Claude has access to five tools
that let it interact with flika programmatically:

| Tool | Description |
|---|---|
| `execute_code` | Run Python code in flika's script namespace (numpy, scipy, pyqtgraph, and all flika process functions are pre-imported). Requires a brief `explanation` shown to the user. |
| `get_window_info` | Inspect the current window: name, shape, dtype, value range, ROI count, metadata keys, current frame. |
| `list_windows` | List all open windows with names, shapes, dtypes, and which is selected. |
| `get_image_stats` | Detailed statistics: min, max, mean, std, median, dimensions description, and a 10-bin histogram. |
| `get_action_log` | Retrieve the session action log (all actions with timestamps, code, and results). Accepts optional `last_n` parameter. |

### Usage

1. Open **AI > Claude > Live Session**
2. Type a request in natural language (e.g., "Apply a Gaussian blur with sigma 3
   and then threshold the result at 150")
3. Claude analyzes your request, calls tools as needed, and reports results
4. You can give follow-up instructions to refine the workflow
5. Original windows are preserved (`keepSourceWindow=True` by default)

### Action Log

Every action taken during a Live Session is recorded with timestamps, the code
executed, the explanation, and the result or error. The action log serves as a
complete audit trail of the session.

- **Show Log** button opens a popup with the full log. You can copy it to the
  clipboard or save it to a text file.
- **Export as Script** extracts all executed code blocks from the log and opens
  them as a single Python script in the Script Editor. Comments from Claude's
  explanations are included.
- **New Session** clears the conversation history, action log, and session-level
  safety approvals.

### Safety

All code generated by Claude passes through the centralized safety checker
(see [AI Safety Settings](#ai-safety-settings) below). If a code block contains
patterns that violate the current safety policy, it is **blocked** before
execution and the user is prompted to approve or deny. Approved patterns are
remembered for the remainder of the session.

The safety policy summary is injected into Claude's system prompt so the model
knows what it should not attempt.

### Examples

```
"Apply a median filter, then threshold, then count connected components"
"What's the mean intensity of the current image?"
"Open all TIFF files in /path/to/folder"
"Detect bright spots and measure their areas"
"Run Richardson-Lucy deconvolution with 20 iterations"
"Compare the histograms of the two open windows"
```

### Interface Details

- Dark-themed chat display with color-coded messages (user, Claude, actions,
  code, results, errors)
- Context label showing current window name, shape, dtype, and number of open
  windows
- Ctrl+Enter keyboard shortcut to send messages
- Collapsible instructions panel
- Stop button to abort a running operation
- Status bar showing the active model and mode

## Claude Script Assistant

**AI > Claude > Generate Script**

A multi-turn conversational assistant for generating Python scripts that work
with flika. Unlike Live Session, the Script Assistant generates code for you to
review, edit, and run -- it does not execute anything automatically.

### Features

- **Multi-turn conversation**: Full message history is maintained so you can
  iteratively refine scripts. Ask follow-up questions like "Now add a threshold
  step" or "Change the sigma to 5".
- **Context-aware**: On the first message, the assistant automatically receives
  the current window's shape, dtype, value range, and the number of open windows.
- **Dark-themed chat display**: Messages are color-coded (user in blue, assistant
  code in orange, errors in red).
- **Collapsible instructions panel** with example prompts.
- **Ctrl+Enter** shortcut to send messages.

### Action Buttons

| Button | Action |
|---|---|
| **Insert into Script Editor** | Opens a new tab in the Script Editor with the generated code for review and editing. |
| **Run Script** | Inserts the code and immediately executes it. Passes through the safety checker first -- if flagged patterns are found, an approval dialog appears. |
| **Copy to Clipboard** | Copies the most recent generated code. |
| **New Conversation** | Clears conversation history and resets the UI. |

### Safety

When you click **Run Script**, the generated code passes through
`check_code_safety()` from the centralized safety module. If blocked patterns are
detected, an approval dialog is shown before execution proceeds.

## Claude Plugin Generator

**AI > Claude > Generate Plugin**

Generate complete, installable flika plugins from natural-language descriptions.
The generator produces a `BaseProcess` subclass with GUI elements (sliders,
checkboxes, combo boxes), validates the result, and can install it immediately.

### Workflow

1. Enter a **plugin name** (e.g., `adaptive_filter`)
2. Describe the plugin's functionality
3. Click **Generate Plugin** (or Ctrl+Enter)
4. Review the code in the syntax-highlighted preview pane
5. Click **Save & Load Plugin** to install it immediately

### Code Preview and Syntax Highlighting

The generated code is displayed in a read-only editor with Python syntax
highlighting (using flika's built-in `PythonHighlighter`). You can review the
code before saving.

### Structural Validation

The generator validates every plugin against critical structural rules using
AST analysis:

| Check | What It Verifies |
|---|---|
| Module-level instance | A line like `my_plugin = MyPlugin()` exists at module level (required for the plugin to load) |
| `gui_reset()` | The `gui()` method calls `self.gui_reset()` |
| `super().gui()` | The `gui()` method calls `super().gui()` |
| `self.start()` | The `__call__()` method calls `self.start(keepSourceWindow)` (or `self.start()` for `BaseProcess_noPriorWindow`) |
| `self.end()` | The `__call__()` method calls `self.end()` and returns its result |

### Auto-Fix

If validation finds certain missing elements, the generator automatically fixes
them:

- **Missing module-level instance**: Appends `snake_case_name = ClassName()` at
  the end of the file. The CamelCase class name is converted to snake_case
  (e.g., `AdaptiveFilter` becomes `adaptive_filter = AdaptiveFilter()`).
- **Missing `self.start()`**: Inserts `self.start(keepSourceWindow)` as the first
  line of `__call__()`, with correct indentation detected from the existing code.

### Plugin Guidelines

If the file `~/.FLIKA/plugin_guidelines.md` exists, its contents are loaded and
injected into the AI system prompt. This lets you define project-specific coding
conventions that the AI will follow.

### Markdown Fence Stripping

If the AI model wraps its output in ` ```python ... ``` ` markdown fences, they
are automatically stripped before validation.

### Save & Load

Clicking **Save & Load Plugin** performs these steps:

1. Saves the plugin code to `~/.FLIKA/plugins/<name>/<name>.py`
2. Creates supporting files:
   - `__init__.py` -- re-exports the plugin module
   - `info.xml` -- menu layout, version, author, date
   - `about.html` -- plugin description page
3. Triggers a plugin reload via `Load_Local_Plugins_Thread` so the plugin
   appears in the **Plugins** menu immediately (no restart required)

### Push to GitHub

After saving, the **Push to GitHub** button becomes available. It:

1. Initializes a git repository in the plugin directory (if not already present)
2. Commits all plugin files
3. Pushes to the configured plugin repository (default:
   `https://github.com/gddickinson/flika_plugins`)

The plugin name is sanitized to prevent shell injection. You may need to
configure git credentials if pushing for the first time.

### Action Buttons

| Button | Action |
|---|---|
| **Generate Plugin** | Generate code from the description |
| **Save & Load Plugin** | Save files and reload plugins |
| **Open in Script Editor** | Open the code for manual editing |
| **Regenerate** | Re-generate with the same description |
| **Copy to Clipboard** | Copy the generated code |
| **Push to GitHub** | Push saved plugin to the repository |

## AI Safety Settings

**AI > Claude > Safety Settings**

A centralized safety layer that controls what AI-generated code is allowed to do.
These settings apply to **Live Session**, **Script Assistant**, and **Plugin
Generator**. The policy summary is injected into all AI system prompts so the
model knows its restrictions.

### Permissions

| Permission | Default | What It Controls |
|---|---|---|
| **File reads** | Allowed | `open('r')`, reading files |
| **File writes** | Blocked | `open('w')`, `os.remove()`, `shutil.rmtree()`, `os.makedirs()`, etc. |
| **Network access** | Blocked | `urllib`, `requests`, `httpx`, `socket`, `http.client` |
| **Subprocess/shell** | Blocked | `subprocess`, `os.system()`, `os.popen()`, `os.exec*()`, `os.spawn*()` |
| **Low-level OS access** | Blocked | `eval()`, `exec()`, `__import__()`, `ctypes`, `os.environ` |

### How Safety Checking Works

1. AI-generated code is scanned against regex patterns grouped by category
2. If a pattern matches a blocked category, the code is **flagged**
3. If the approval dialog is enabled, the user sees a warning listing the
   specific flagged operations and can approve or deny
4. Approved patterns are added to a **session-level whitelist** so the user is
   not asked again for the same pattern during the current session
5. The whitelist is cleared when safety settings are changed or when a new
   session is started

### Allowed Directories

An optional comma-separated list of directories the AI may access. If left
empty, only flika's own directories are permitted.

### Approval Dialog

When enabled (default), an approval dialog appears before running any flagged
code. The dialog shows:

- The specific dangerous patterns detected (e.g., "subprocess module",
  "os.remove()")
- Options to approve (adds patterns to session whitelist) or deny

The approval dialog can be disabled in the settings, in which case all flagged
code is silently blocked.

### Important Disclaimer

These are **policy restrictions**, not a sandbox. They work by pattern-matching
against known dangerous operations and instructing the AI model to respect them.
A determined or creative prompt could potentially bypass these checks. Always
review AI-generated code before running it, especially if you have enabled
permissive settings. This warning is displayed prominently in the Safety Settings
dialog.

### Settings Persistence

All safety settings are stored in `g.settings` (persisted to
`~/.FLIKA/settings.json`) and take effect immediately when saved.

## AI Denoiser

The denoiser uses deep learning models to remove noise from microscopy images
while preserving structural details.

### Usage

1. **AI > AI Denoiser** to open the dialog
2. Select a denoising model (or use the default)
3. Configure parameters (patch size, overlap)
4. Click Apply

The denoiser processes the current window and creates a new window with the
denoised result.

### Supported Models

Models are stored in `~/.FLIKA/models/` and can be downloaded from the Model Zoo.

## Pixel Classifier

Train a machine learning classifier to label pixels based on user-drawn examples.

### Workflow

1. Open the image to classify
2. **AI > Pixel Classifier** to open the classifier dialog
3. Draw ROIs on examples of each class (e.g., "cell", "background", "artifact")
4. Assign class labels to each ROI
5. Click **Train** -- the classifier learns from the labeled pixels
6. Click **Predict** -- applies the trained model to the full image

### Features Used

The pixel classifier extracts multi-scale features at each pixel:

- Gaussian blur at multiple sigmas
- Gradient magnitude
- Laplacian of Gaussian
- Structure tensor eigenvalues

The resulting feature vector is classified using the trained model.

```python
# Access the pixel classifier programmatically
g.m._ai_classify()
```

## Particle Localizer

AI-based particle detection that learns from user annotations.

### Workflow

1. Open a stack with particles
2. **AI > Particle Localizer** to open the dialog

4. Train the localizer

[Process > Detection](process_menu.md#detection) and integrates with the

## Segmentation

### Cellpose

[Cellpose](https://www.cellpose.org/) is a deep learning model for cell
segmentation.

1. **AI > Segmentation > Cellpose**
2. Select a pre-trained model:
   - `cyto` -- general cytoplasm segmentation
   - `cyto2` -- improved cytoplasm model
   - `nuclei` -- nuclear segmentation
   - Custom trained models
3. Set the expected cell diameter
4. Run segmentation

The result is a labeled image where each cell has a unique integer ID.

**Requires:** `pip install cellpose`

### StarDist

[StarDist](https://github.com/stardist/stardist) detects cells using
star-convex polygons. Best suited for round or convex-shaped nuclei.

1. **AI > Segmentation > StarDist**
2. Select a pre-trained model:
   - `2D_versatile_fluo` -- fluorescence microscopy
   - `2D_versatile_he` -- H&E stained histology
   - Custom models
3. Configure probability and NMS thresholds
4. Run detection

**Requires:** `pip install stardist`

### SAM Interactive Segmentation

The [Segment Anything Model (SAM)](https://segment-anything.com/) provides
interactive segmentation guided by user clicks.

#### Workflow

1. **AI > Segmentation > SAM Interactive**
2. The SAM dialog opens with the current image
3. **Click** on an object to generate a segmentation mask
4. **Shift+Click** to add positive prompts (include this region)
5. **Ctrl+Click** to add negative prompts (exclude this region)
6. The mask updates in real-time
7. Accept the mask to create a binary window

SAM requires downloading the model weights on first use. Weights are cached in
`~/.FLIKA/models/`.

**Requires:** `pip install segment-anything` and a SAM checkpoint

## API Key Configuration

The Anthropic API key required for all Claude-powered features (Live Session,
Script Assistant, Plugin Generator) is stored securely via the system keyring
(macOS Keychain / Windows Credential Manager / Linux Secret Service). The key is
**never** stored in plaintext in `settings.json`.

- Set the key in **File > Settings** (API Key field)
- Delete the key with the **Delete API Key** button
- Legacy plaintext keys are automatically migrated to the keyring on first access
- The key is never included in exported settings or provenance files
- Alternatively, set the `ANTHROPIC_API_KEY` environment variable

### Model Selection

The AI model can be configured in **File > Settings** (default:
`claude-sonnet-4-20250514`). This setting applies to all Claude-powered
features.

## GPU Acceleration

AI tools benefit significantly from GPU acceleration. Check your GPU status via
**Help > GPU/Acceleration Status**.

- CUDA GPUs (NVIDIA) provide the best performance for PyTorch-based tools
- MPS (Apple Silicon) is supported for many operations
- CPU fallback is always available

Configure the acceleration device in **File > Settings**:

- `Auto` -- automatically select the best available device
- `CPU` -- force CPU computation
- `CUDA` -- use NVIDIA GPU
- `MPS` -- use Apple Silicon GPU

```python
# Check current device
from flika.utils.accel import get_device
print(get_device())  # e.g., 'cuda:0', 'mps', 'cpu'
```

## Microscopy Simulation

**Simulation > Simulation Builder...**

Flika includes a comprehensive microscopy simulation engine for generating
synthetic microscopy data with known ground truth. This is useful for algorithm
validation, benchmarking, training data generation, and teaching.

### Simulation Builder Dialog

The Simulation Builder is a 4-tab dialog:

| Tab | Contents |
|---|---|
| **Quick Start** | Preset selector with description. Click "Customize..." to modify parameters. |
| **Sample** | Structure type, fluorophore preset, dynamics (motion type, diffusion coefficient, bleaching, blinking) |
| **Microscope** | Modality, optics (NA, wavelength, PSF model), camera settings (type, pixel size, QE, noise) |
| **Output** | Dimensions (width, height, Z slices, time frames, pixel size), background level, ground truth toggle, window name |

### Quick Presets

The **Simulation > Quick Presets** submenu provides 10 ready-to-use
configurations:

| Preset | Modality | Structure | Key Parameters |
|---|---|---|---|
| Beads - PSF Calibration | Widefield | 100 beads | 256x256, static |
| TIRF - Single Molecules | TIRF | 200 beads | 512x512, 100 frames, EMCCD |
| Confocal - Cell with Organelles | Confocal | Cell + organelles | 512x512x30, 80nm pixel |
| PALM/STORM - Dense Filaments | SMLM | 20 filaments | 256x256, 5000 frames |
| DNA-PAINT - Grid Pattern | DNA-PAINT | 100 beads | 256x256, 2000 frames |
| Light-Sheet - Cell Spheroid | Light-sheet | 10-cell field | 256x256x50 |
| FLIM - Two Lifetimes | FLIM | Cell | 128x128, 256 time bins |
| Calcium - Neuron Spikes | Widefield | Neuron | 128x128, 1000 frames |
| SIM - Resolution Target | SIM | 15 filaments | 256x256, 50nm pixel |
| Tracking - Brownian Particles | Widefield | 50 beads | 256x256, 500 frames, D=0.1 |

### Simulation Pipeline

The `SimulationEngine` follows a composable pipeline:

```
Structure -> Fluorophore Labeling -> PSF Convolution -> Dynamics ->
Optical Modality -> Camera/Noise Model -> Output
```

#### Structure Types

- **Beads**: Random point emitters (configurable count and radius)
- **Filaments**: Worm-like chains with adjustable persistence length and thickness
- **Cells**: Single cell masks (round, elongated, irregular, neuron), optionally
  with organelles (nucleus + mitochondria)
- **Cell field**: Multiple cells arranged in a field

#### PSF Models

- **Gaussian**: Fast analytical approximation
- **Airy**: 2D Airy disk pattern
- **Born-Wolf**: Scalar diffraction-based 3D PSF
- **Vectorial**: Full vectorial PSF with polarization effects

#### Microscopy Modalities

- **Widefield**: Standard epifluorescence
- **TIRF**: Total internal reflection fluorescence (configurable penetration depth)
- **Confocal**: Optical sectioning (configurable pinhole size in AU)
- **Light-sheet**: Selective plane illumination (configurable sheet thickness)
- **SMLM**: PALM/STORM single-molecule localization (density-controlled blinking)
- **DNA-PAINT**: Transient binding kinetics (k_on, k_off, imager concentration)
- **FLIM**: Fluorescence lifetime imaging (time-binned histograms)
- **SIM**: Structured illumination microscopy (pattern generation + reconstruction)
- **STED**: Stimulated emission depletion

#### Camera Models

Three camera types with realistic noise:

| Parameter | CCD | EMCCD | sCMOS |
|---|---|---|---|
| Read noise (e-) | ~5 | ~0.5 | ~1.5 |
| EM gain | N/A | 1-1000 | N/A |
| Dark current | Low | Low | Very low |
| QE | 0.5-0.7 | 0.9-0.95 | 0.7-0.82 |

Additional camera parameters: pixel size, gain (ADU/e-), baseline offset, bit
depth, exposure time.

#### Dynamics

- **Static**: No motion
- **Brownian**: Random walk with diffusion coefficient D
- **Directed**: Brownian + constant velocity
- **Confined**: Brownian within a confinement radius
- **Anomalous**: Sub/super-diffusion with exponent alpha

Additional dynamic phenomena:
- **Bleaching**: Exponential photobleaching
- **Blinking**: On/off rate stochastic switching
- **Calcium transients**: Spike trains with configurable rate and decay

#### Fluorophore Presets

10 built-in fluorophore configurations: EGFP, mCherry, Alexa488, Alexa647,
Cy3, Cy5, Atto655, DAPI, GCaMP6f, tdTomato. Each preset defines photon output,
lifetime, blinking rates, and bleaching rate.

### Ground Truth

When "Include ground truth" is checked (default), the simulation stores detailed
ground truth in `window.metadata['simulation']`:

| Key | Contents |
|---|---|
| `config` | The full `SimulationConfig` dataclass |
| `structure_mask` | The base structure image |
| `binary_mask` | Binary mask of the structure |
| `positions` | Emitter positions (N x ndim array) |
| `positions_per_frame` | Detection table with frame, x, y columns |
| `trajectories` | Particle trajectories (N x T x ndim) |
| `track_dict` | Track dictionary `{id: (N, 3) array}` compatible with SPT |
| `diffusion_coefficients` | True D values per particle |
| `segmentation_mask` | Labeled regions (cell fields only) |
| `n_photons` | Pre-noise photon image |
| `emitter_states` | On/off state matrix (SMLM only) |
| `calcium_trace` | 1D calcium trace (calcium mode only) |
| `calcium_spike_frames` | Frame indices of spike peaks |

### Benchmarking

The simulation package includes an evaluation framework for quantitative
comparison of analysis results against ground truth:

- **Detection metrics**: Precision, recall, F1 score (Hungarian matching)
- **Tracking metrics**: MOTA (Multiple Object Tracking Accuracy), MOTP
  (Multiple Object Tracking Precision)
- **Segmentation metrics**: Jaccard index, Dice coefficient
- **Calcium metrics**: Event detection accuracy

The **Benchmark Dialog** (`Simulation > Benchmarks`) provides a tree-based
benchmark selector, results table, and export options (text and JSON).

### Scripting

```python
# Run a preset from the script editor
from flika.process.simulation import simulate
simulate.run(preset='Tracking - Brownian Particles')

# Or configure manually
from flika.simulation.engine import SimulationConfig, SimulationEngine
from flika.simulation.noise import CameraConfig

config = SimulationConfig(
    nx=256, ny=256, nt=100,
    modality='widefield',
    structure_type='beads',
    structure_params={'n_beads': 50},
    motion_type='brownian',
    motion_params={'D': 0.1},
    camera=CameraConfig(type='sCMOS'),
)
engine = SimulationEngine(config)
stack, ground_truth = engine.run()

# Create a window with ground truth
from flika.window import Window
w = Window(stack, name='Simulation')
w.metadata['simulation'] = ground_truth
```
